PHASE 0 — SECURITY & INFRASTRUCTURE FOUNDATIONS
================================================================================
Do this before writing feature code. Retrofitting security later is how
public sites get breached. Every later phase must comply with this list.

0.1 Authentication: JWT (decision locked in)
GOAL: Stateless, secure token auth suitable for a public site, decoupled
frontend/backend, and future mobile app.
WHERE: backend/accounts/, using djangorestframework +
  djangorestframework-simplejwt (install these two packages)
WHAT TO BUILD:
  - Access token: short-lived (10-15 minutes), sent in the Authorization
    header as "Bearer <token>" on every API request.
  - Refresh token: longer-lived (7 days), stored in an httpOnly, Secure,
    SameSite=Strict cookie — NEVER in localStorage or a JS-readable
    cookie. This is the single most important rule: if the refresh token
    is readable by JavaScript, a cross-site-scripting (XSS) bug becomes
    full account takeover.
  - Access token CAN live in memory (a React context/state variable),
    not localStorage — avoids XSS token theft on page reload attacks.
  - POST /api/auth/token/ (login) -> returns access token in body, sets
    refresh cookie
  - POST /api/auth/token/refresh/ -> reads refresh cookie, returns new
    access token
  - POST /api/auth/token/logout/ -> blacklists the refresh token
    (SimpleJWT's token blacklist app) and clears the cookie
  - Token rotation: enable ROTATE_REFRESH_TOKENS + BLACKLIST_AFTER_ROTATION
    in SimpleJWT settings so a stolen refresh token can't be replayed
    indefinitely.
  - Admin (is_staff) users use the SAME JWT system, just with an
    additional permission check (IsAdminUser) on admin-only endpoints —
    do not build a second parallel auth system.
ACCEPTANCE CHECK: login returns an access token + sets an httpOnly
refresh cookie (verify via browser dev tools that JS document.cookie
cannot read it); expired access token auto-refreshes transparently on
the frontend via an axios/fetch interceptor.

0.2 Core Django/web security hardening
GOAL: Standard public-facing web app protections.
WHAT TO BUILD / CONFIGURE:
  - DEBUG = False in production; SECRET_KEY and all credentials loaded
    from environment variables (use python-decouple or django-environ),
    never committed to git.
  - ALLOWED_HOSTS restricted to your real domain(s).
  - Force HTTPS: SECURE_SSL_REDIRECT = True, SESSION_COOKIE_SECURE = True,
    CSRF_COOKIE_SECURE = True (once you have a TLS certificate — e.g. via
    Let's Encrypt on whatever host you deploy to).
  - CORS: use django-cors-headers, explicitly whitelist only your own
    frontend domain(s) — never CORS_ALLOW_ALL_ORIGINS = True.
  - CSRF protection stays on for any cookie-based endpoints (the JWT
    refresh-cookie endpoint included); Django's CSRF middleware handles
    this, just don't disable it.
  - Password storage: rely on Django's default PBKDF2/Argon2 hasher —
    never store or log plaintext passwords.
  - Rate limiting: use django-ratelimit (or DRF throttling classes) on
    login, register, and password-reset endpoints — e.g. 5 attempts per
    minute per IP — to block brute-force attacks.
  - Input validation: use DRF serializers for every endpoint (never trust
    raw request.POST/body) so malformed or malicious payloads are
    rejected before touching the database.
  - SQL injection: Django ORM already parameterizes queries by default —
    just make sure no code ever uses raw SQL string concatenation.
  - XSS: React escapes output by default — never use
    dangerouslySetInnerHTML with unsanitized user input (e.g. review
    text, product descriptions from admin). If rich text is ever needed,
    sanitize with a library like DOMPurify first.
  - File/image uploads (bulk image upload, review photos): validate file
    type + size server-side (not just by extension — check actual MIME
    type), store outside of any executable path, and serve via a
    dedicated media path.
  - Security headers: add django-csp or manual middleware for
    Content-Security-Policy, X-Content-Type-Options: nosniff,
    X-Frame-Options: DENY (prevents clickjacking on login/checkout pages).
  - Admin panel: change the default /admin/ URL to something non-obvious
    in production settings, enforce strong passwords, and enable Django's
    admin login rate limiting.
  - Dependency hygiene: run pip list --outdated and npm audit
    periodically; pin versions in requirements.txt/package.json.
ACCEPTANCE CHECK: a checklist file (security_checklist.md) exists listing
every item above with a checkbox; all boxes checked before Phase 10 sign-off.
